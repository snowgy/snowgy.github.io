Note: Schema for the schemaless

Posted on 2019-06-19 | In research |

本文主要讲述了AsterixDB如何做内存压缩，以及压缩后access data的方法和优化。

Motivation

Open fields bring storage overhead which will affect the query efficiency. Having an automated mechanism to infer and consolidate the schema information for the ingested records without losing the flexibility and the experience of the schema-less stores is desirable.

Question:

  1. Why open fields will bring storage overhead?

    Because we need to save some common components repeatly.

  2. Why the storage overhead will reduce the query efficiency?

    Bigger data blocks, bigger I/O cost

Apache AsterixDB Architecture

LSM-Based Schema Inference And Tuple Compaction Framework

Main idea factor out the fields’ metadata from their values for each record and construct the overall schema during the LSM-flush operation.

Tuple Compactor the module that infers the schema and compacts the record

Framework Overview

The newly inferred schema is a super-set(or union) of all the previously inferred schemas.

Given this property, we only need to store the most recent schema of all the mergeable components as it covers the fields of all the previously flushed components.

pros the merge operation does not need to block the modifications on the in-memory schema while flushing the in-memory

Schema Structure

The schema stucture describes the tree structure of the ingester JSON-like records, where the inner nodes of the tree structure of the tree describes the different

Schema Structure Subcomponents

  • A tree-structured summarizes the ADM structure of the ingested records.

  • A dictionary that encodes the field names into IDs

  • An Inverted-index that stores the hash-codes of field names and their associated IDs in the dictionary

Schema Structure Maintenance

anti-matter tuple when deleting a tuple, construct an anti-matter tuple and insert it into the in-memory component.

anti-matter schema Exploit the lookup operation of the delete to retrieve the schema of the deleted tuple and include it within the anti-matter tuple.The schema in the anti-matter tuple is anti-matter schema.

Tuple Compaction

AsterixDB closed type is suitable for fixed schema but not a dynamic one.

Sparse tuples can incur a storage overhead as the closed type tuples need to maintain a bitmap which indicates whether a filed value is present, null or missing.

Vector-based Physical Data Format

Accessing Values in the Vector-based Tuples

Each value in the JSON or ADM document has a unique path that starts from the root. In AsterixDB, a value is accessed by providing the path expression that describes the value’s location.

Compacting Vector-based Tuples

Query Processing

Handling Heterogeneous Schemas

  1. each of the distributed partition executes the same job.
  2. No coordination

Therefore, the schema in each partition can be different from other schemas in other partitions.

Solution 1

Uncompact each tuple into its original form by including the field names rather than the field name IDs.

Drawback

Field name size is relatively large which increases the total tuple size. Larger tuple size means a smaller number of tuples per batch which implies more memory copy operations.

Solution 2

Distribute the schema information of each partition to all nodes in the cluster at the beginning of the query execution.

Each node receives and caches each partition’s schema information along with its partition ID and serves them to each of its executors.

To minimize the distribution cost, we only distribute the schema information when 1) the query plan contains a non-local exchange operator such as hash-partition-exchange and 2) the resulting tuples, after the non-local exchange connector, contains object fields.

Processing Compacted Tuples

Vector-based format: linear scan

ADM physical format: logarithmic time

Thus delay field access expression is somtimes costly for Vector-based format.

solution

Added one rewrite rule to the AsterixDB query optimizer to consolidate field access expressions in the plan into a single expresssion.

We also take the opportunity to push the consolidated field access expression down through the scan operator and project only the needed values instead of the whole tuple.
